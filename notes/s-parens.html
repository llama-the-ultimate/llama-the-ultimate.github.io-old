S (and parentheses)

In the SKI combinator calculus we have three functions. S, K and I:

I x = x
K x y = x
S x y z = x z (y z)

You can translate untyped lambda calculus terms to SKI by following some rules pretty mechanically. Anyway the I and K functions are reasonably straightforward. The S is funny.

You use the S when there’s a lambda abstraction that contains one function application:

λx.F A = S (λx.F) (λx.A)

(Where F and A are arbitrary terms (although if x does not appear free in either F or A you can maybe use a simpler rule for translating and get rid of the x right away). You can then go on and translate λx.F and λx.A by following the same set of rules, if you wanna.)

What with definition of S earlier, all of this should be fine:

S x y z = x z (y z)
λx.F A = S (λx.F) (λx.A)
S (λx.F) (λx.A) z = (λx.F) z ((λx.A) z)
λx.F A = (λx.F) z ((λx.A) z)

So, we kind of turn the function and the argument of the inner application into function-function and argument-function. Now if we apply S (λx.F) (λx.A) to foo, S will pass foo (its z) into λx.F and λx.A. All xes in F and A will be replaced with foo. Same as if we applied λx.F A to foo.

The x z (y z) bit used to trip me up. It kind of looks like it passes two arguments, z and y z, to x. And then why does the first argument appear in the second and and ??? I think it became clearer to me when I saw it written with more parentheses:

S x y z = (x z) (y z)

x is the function-function and y is the argument-function. They get one z each, and the result-function is applied to the result-argument.

(Also helped to run into B and C. They’re like S, but B only passes its z the the argument-function and C only passes it to the function-function:

S x y z = x z (y z)
B x y z = x (y z)
C x y z = x z y

Again, with more parentheses:

S x y z = (x z) (y z)
B x y z = x (y z)
C x y z = (x z) y

Like that it’s easier for me to see how they are different, which parts are the same, …)
